## 一、定位与目标

### 1.1 一句话定位

把简历里的三个项目（自适应学习 Agent、大模型评测系统、路径规划 Transformer）变成面试场上**可背诵、可推演、可溯源**的答辩剧本。

### 1.2 与已有飞书文档的分工

| 文档 | 定位 | 回答的问题 |
|-|-|-|
| 《技能树 Agent 实现详解》(T9D6...) | 代码级实现文档 | "怎么做的" |
| 《大厂 AI Agent 实习备战手册》(L74C...) | 通用知识体系 | "Agent 领域该懂什么" |
| **本文（新）** | **面试应答手册** | **"面试官怎么问、我怎么答"** |

边界：不重复上述两文档的代码细节和通用知识，只在追问话术里用链接引用两者。聚焦"临场应答"这个空白地带。

### 1.3 成功标准

- 面试前能当背诵手册通读，每个项目的核心问答能脱稿复述
- 面试中被深挖任意项目时，能快速定位到对应 Q&A 并流畅作答
- 被追问"这个数据怎么来的"时，能立刻调出溯源卡给出可信回答
- 被问及 2026 前沿（MCP/A2A/推理模型）时，有准备好的应答而非临场拼凑

### 设计要点

- 第 1-3 章是主体，结构完全统一（四段式：定位→Q&A→深挖→溯源），背诵和推演节奏一致
- 第 4 章收口跨项目题，防止"单项目会讲、串起来卡壳"
- 第 5 章前沿只补 4 个精准方向（不是百科全书），每个方向都配"面试官可能怎么追问 + 你怎么接"
- 第 6 章是面试首尾两端的高频环节，独立成章

### 各章格式约定

| 章 | 格式 |
|-|-|
| 第 1-3 章 | 统一四段式（见第四章规格） |
| 第 4 章跨项目串联题 | 每题沿用 Q&A 三段式（❓问题 → 💡要点 → 🔗关联），但要点需跨两个以上项目 |
| 第 5 章前沿补充 | 每个方向固定四段：现状速记 → 关键数据/特性 → 与项目的关联 → 追问应答（❓💡格式） |
| 第 6 章自我介绍+反问 | 自我介绍给 1 分钟/2 分钟两个模板（口语稿）；反问环节列 5 个问题并各配一句"为什么这个问题有深度" |

## 四、四段式内容规格

第 1-3 章每章固定四段，格式统一，确保临场检索一致。

### ① 一句话定位 + 架构速记

- **一句话定位**: 用 1-2 句话说清"这项目是什么、解决什么痛点、我负责什么"。面试官问"介绍一下这个项目"时直接背。
- **架构速记**: 一个极简文字流程图（3-5 行），面试时能在白板/纸上快速画出来。

示例（第 1 章 自适应学习 Agent）：
```
用户输入 -> Planner(意图分流) -> chat? 短路直答
                                -> Executor(ReAct循环, max=6, +Reflexion封顶1轮)
                                -> Writer(needs_doc? 三模板产出飞书文档)
```

### ② 高频追问 Q&A

每题统一三段式：

```
❓ 面试官问题
💡 回答要点（3-5 个 bullet，可背诵，口语化但不口水）
🔗 关联（引用文档 A/B 的章节或代码位置）
```

- **题目来源**: 文档 B 已有的追问（如 ReAct vs CoT、Reflexion 会不会无限循环）+ 根据简历新增的深挖题（如"5 家供应商具体是哪几家""BM25 的 k1/b 参数怎么定的"）
- **话术风格**: 口语化但不口水，每个 bullet 是一个完整的论点而非关键词堆砌

### ③ 技术深挖要点

- 不重复文档 A 的代码，而是提炼"面试官最可能往下挖的技术点 + 你该讲到什么深度"
- 每个要点格式: `技术点 -> 一句话本质 -> 如果追问往哪展开`
- 示例: `BM25 -> IDF 加权的关键词检索 -> 追问可展开 k1/b 参数含义、与纯向量的互补（精确匹配 vs 语义）`

### ④ 数据溯源卡

表格形式，每个量化数字一行：

| 简历数字 | 怎么测的 | 在哪复现 | 口径限制 |
|-|-|-|-|
| Top-5 召回 93% | 15 条带 ground-truth 的 query 跑纯向量通道 | `eval/run_rag_eval.py` | 纯向量通道，三路融合上线后更高；简历写的 60 条标注 query 是另一口径 |

- **作用**: 面试官问"这个 93% 怎么来的"，直接照着卡片答"测了多少条、什么口径、在哪复现、有什么局限"——主动说局限反而显可信。

## 五、内容来源与可验证性约束

### 5.1 一手来源（最高可信度，面试可直接说"我做的"）

- **简历素材**: `resume/shared/experience/*.tex` 里的量化数字和描述
- **文档 A《技能树 Agent 实现详解》**: Agent 项目的实现细节、实测数据（13 条黄金用例、134 个 pytest、15 条 RAG query）
- **代码注释**: 各 `.tex` 文件头的"数据口径/来源"注释

### 5.2 二手来源（需标注"业界共识"而非"我做的"）

- **文档 B《备战手册》**: 通用知识（ReAct 论文、RAG 流程、框架对比）——用作话术的背景知识，不当作项目经历
- **2026 前沿搜索结果**: MCP/A2A/Agent 评估/推理模型——在第 5 章使用，明确标注是"业界动态"，回答时用"我了解到..."而非"我做过..."

### 5.3 红线

- **不编造**: 如果某个追问的答案在已有素材里找不到，Q&A 里标 `[待你补充：xxx]`，不自行虚构
- **数字对齐**: 简历数字与溯源卡完全对齐
- **项目 3 特别处理**: 路径规划 Transformer 在文档 A/B 中没有覆盖，深挖要点只能从 `.tex` 素材 + 代码注释推导，凡涉及实现细节且无法验证的一律标 `[需确认]`

## 六、2026 前沿补充内容（第 5 章素材）

以下为搜索验证后的前沿素材，写入第 5 章时每个方向配"面试官可能怎么追问 + 你怎么接"。

### 6.1 MCP 协议 2026 新进展

- 规范最新版本 **2025-11-25**（上一版 2025-06-18），基于 TypeScript schema 定义
- **Streamable HTTP transport** 取代旧 HTTP+SSE（服务器可主动断连支持轮询）
- **OAuth 2.1 授权**：MUST 实现 RFC 9728 Protected Resource Metadata + OIDC Discovery 1.0、Client ID Metadata Documents、增量 scope consent
- 实验性 **Tasks**（durable 请求 + 轮询/延迟取回）、Sampling 支持工具调用、JSON Schema 2020-12 默认方言
- 注意：规范是 OAuth 2.1 access token，**并非 JWT 原生**
- 主要采纳方：Anthropic（Claude 4 API 内置 MCP connector）、GitHub Copilot、Cursor、Replit、Block、Cognition
- **面试追问点**: Streamable HTTP 为何取代 HTTP+SSE、MCP 三角色（Host/Client/Server）职责、OAuth discovery 流程、JSON-RPC 2.0 消息格式
- **与项目的关联**: 你在评测系统里封装了 Unlimited-OCR 为 MCP Server 暴露 6 个 OCR 工具接口，可结合讲"为什么选 MCP 而非直接 API"

### 6.2 A2A 协议

- Google 于 2025 年 4 月推出的开放协议，定位 **agent-to-agent 通信标准**
- 与 MCP **互补而非竞争**：MCP 管 agent-to-tool（agent 连接工具/数据源），A2A 管 agent-to-agent（不同框架、不同公司的 opaque agent 之间协作）
- 技术上基于 **JSON-RPC 2.0 over HTTP(S)**，核心概念是 **"Agent Card"**（能力发现）
- 支持同步/流式(SSE)/异步推送三种交互，强调**不暴露 agent 内部状态/记忆/工具**
- 2026 生态已成型：官方提供 Python/Go/JS/Java/.NET/Rust 六种 SDK，DeepLearning.AI 已上线配套课程，与 Google ADK、LangGraph、BeeAI 等框架集成
- **面试追问点**: "A2A 与 MCP 分层互补"这一关键关系；为什么 agent 间通信不能直接用 MCP

### 6.3 Agent 评估 2026

- **SWE-bench Verified 趋饱和**: 2025 年底 live-SWE-agent + Claude 4.5 Opus 达 79.2%，2026 年 2 月最新一批（Claude 4.5 Opus / Gemini 3 Flash / MiniMax M2.5）稳定在 75-77%，mini-SWE-agent 仅 100 行代码就达 74%，顶部分数逼近天花板
- 研究重心转向 **SWE-bench Multimodal**（带图像的问题）和 **Multilingual**
- **tau-bench 已演进到 τ³-bench**: 新增 banking_knowledge 知识检索域、语音全双工（OpenAI/Gemini/xAI 实时音频）评测模态，并修复 75+ 任务缺陷；TAU-bench 步数上限从 30 提升到 100
- **AgentBench** 2025 年 10 月推出 **FC（function-calling）版**并集成 **AgentRL** 强化学习框架
- GAIA 仍为通用助手基准
- 2026 新趋势："饱和基准被升级（多模态/语音/知识），并强调可复现容器化评测"
- **与项目的关联**: 你的评测系统聚合了 36 个 Benchmark（含 MMLU-Pro、SWE-Bench、AIME），可结合讲"你用这些 benchmark 做横向评测时，怎么处理饱和问题和量纲不一"

### 6.4 推理模型对 Agent 的冲击

- **Claude 4 混合推理**: 两种模式（即时响应 + Extended Thinking）。核心新机制 **Extended Thinking with Tool Use（beta）**: 模型可在思考链中交替进行推理与工具调用（如 web search），打破"先想完再调工具"的限制
- 配套能力：并行工具执行、**Memory Files**（本地文件存关键信息维持长程连续性，可跑数小时/数千步）、**Thinking Summaries**（小模型压缩长思维链，仅约 5% 场景需要，Developer Mode 可保留原始 CoT）
- 对 Agent 架构影响：支持数小时级长任务、多步轨迹、API 新增 code execution 工具 + MCP connector + Files API + 1 小时 prompt 缓存
- **新共识**:
  1. 推理模型内置 CoT，与 ReAct 的 Thought 环节重叠，单步推理已无需显式写 Thought
  2. 但 ReAct 的核心价值是 Thought→Action→Observation 的多轮循环与外部环境交互。推理模型只解决"想"，不解决"用工具取新信息"和"执行"，多步、有外部状态依赖的任务仍需 Agent loop
  3. 推理模型 + 原生 function calling 使显式 ReAct 提示式微（模型内部推理后直接调工具），但 Agentic loop 仍是多步任务必需，并非被淘汰
  4. 长上下文不取代 RAG：超长上下文检索精度下降、成本高、覆盖不了全量语料、难以及时更新、缺可溯源引用。共识是混合架构——长上下文做单文档/少样本深度推理，RAG 负责大规模召回、新鲜数据与可引用
- **面试一句话答法**: 推理模型让 ReAct 的"Thought 提示"退场，但没让 Agent 循环退场；长上下文压缩了 RAG 的适用面，但没取代 RAG
- **与项目的关联**: 你的 Agent 用的是 ReAct 文本 + Reflexion，可结合讲"如果换 Claude 4 的 extended thinking，Reflexion 这一层可能被原生 thinking 消化，但多工具循环和写操作提案闭环仍然需要"

## 七、实施方式

### 7.1 创建飞书文档

- **工具**: `lark-cli docs +create`（XML 格式，支持 callout/grid/table 等富 block）
- **读取参考文档**: 实施前必读
  - `references/lark-doc-xml.md`（XML 语法规则）
  - `references/style/lark-doc-style.md`（元素选择、丰富度规则、颜色语义）
  - `references/style/lark-doc-create-workflow.md`（从零创建工作流）
- **身份**: `--as user`（文档归属用户本人）
- **预估篇幅**: 中等偏长（6 章，第 1-3 章各约 1500-2000 字，第 4-6 章各约 800-1200 字，总计约 10000-13000 字）

### 7.2 富文本元素使用约定

| 元素 | 用途 |
|-|-|
| `<callout emoji="❓">` | Q&A 里的面试官问题 |
| `<callout emoji="💡">` | Q&A 里的回答要点 / 设计要点 |
| `<callout emoji="⚠️">` | 红线 / 易踩坑提醒 |
| `<callout emoji="🔗">` | 关联引用（文档 A/B 链接） |
| `<table>` | 数据溯源卡、架构对比 |
| 代码块 | 架构速记流程图、伪代码 |
| `<bullet>` | Q&A 回答要点列表 |

### 7.3 实施顺序

按章顺序创建（0→1→2→3→4→5→6），每章创建后可单独 review。由于是新建文档，用 `docs +create` 一次性写入全部内容，或分章 `docs +update --command append` 追加（若内容过长导致单次写入超限）。

## 八、风险与缓解

| 风险 | 缓解 |
|-|-|
| 文档过长导致创建失败 | 分章 append，或先创建框架再逐章填充 |
| 项目 3（路径规划）素材不足，深挖时露馅 | 凡无法验证的细节标 `[需确认]`，Q&A 只覆盖 `.tex` 素材能支撑的题目 |
| 2026 前沿信息有时效性，面试前可能过时 | 第 5 章每个方向标注信息截止日期（2026-07），面试前建议快速复核 |
| 话术过于书面化，背诵后显得不自然 | 话术以口语化为目标，Q&A 要点用"第一人称陈述句"而非"第三人称描述" |
