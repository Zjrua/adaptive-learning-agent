# Skill: resume-editing

求职简历撰写与对外形象管理。简历的读者分三层，写法必须同时满足：

| 读者 | 关心什么 | 怎么满足 |
|------|---------|---------|
| **HR / 非技术背景** | 你做了什么、有什么业务价值、能不能扫读完 | 业务语言、可读结构、无红旗 |
| **技术面试官** | 你怎么做的、技术深度、数据经不经得起追问 | 保留技术方案、可验证指标 |
| **背调 / 仓库审查** | 简历写的和 GitHub 真的对得上吗？项目是真实工程还是面试道具？ | 口径一致、历史干净、无动机暴露 |

第三层是最容易忽视的——面试官会点开你的 GitHub，翻 README、翻 commit history、翻文档措辞。简历写得再好，仓库里露馅就全崩。

---

## 第一部分：简历文本撰写

### 原则 1 — 双层写法（技术方案 + 业务价值配对）

每条要点写成「技术名词 + 紧跟一句业务效果」的配对。HR 只读标签和结尾数据（扫读），面试官读中间技术（细读），同一句话同时服务两类读者。

```
\item \textbf{检索增强}：手写 BM25 + 向量余弦相似度做双路融合检索，Top-5 召回 93%
         ^^^^^ HR懂的标签 ^^^^    ^^^^^技术方案(面试官看)^^^^    ^^^业务结果(都看)^^^
```

规律：
1. **前缀标签用 HR 懂的词**：「检索增强」「可靠性设计」「效率优化」「部署选型」，不要用「双路融合 BM25 IDF 加权」当标签
2. **技术方案放在标签后**：具体技术名词（BM25、ArcFace、MLX）保留，这是给面试官看的
3. **结尾落到业务结果/数据**：召回率、准确率、耗时、提速倍数

### 原则 2 — 绝对值是红旗，必须改写

100% / 0% 这类绝对值，在 HR 和面试官眼里都高度可疑——真实工程几乎不可能绝对。

- `准确率 100%` → `关键边界用例全覆盖` / `在测试集上零误判` / `准确率约 98%`
- `命中率 0%→100%` → `参数从失效区间扫描到拐点，命中率显著提升`

**为什么**：100% 不是强指标，是红旗。HR 看到会怀疑造假；面试官会追问测试集规模，答不上反而暴露。中等偏高的真实数字（93%、0.80）比绝对值更可信。

### 原则 3 — 项目经历标准范式

每个项目：**1 句背景 + 3 条成果，每条 = 动作 + 技术方案 + 可验证结果**。

3 条是信息密度甜点——少了单薄，多了 HR 扫不完。每条必须有「结果」，否则只是罗列技术栈。

### 原则 4 — 删模糊词和弱指标，换可验证指标

**删模糊词**（等于没说）：表现良好、端到端可演示、平滑下线、效果不错、显著提升（无数字）。

**删弱指标**（过程量不是结果量）：代码行数、文件个数、提交次数。

| 弱（删） | 强（换） |
|---------|---------|
| 提速约 4× | 6 worker 跑完 484 页，耗时 115.6 分钟，提速 4.1× |
| 代码约 3165 行 | （删掉，换业务结果） |
| 表现良好 | 在 15 条查询集上 Top-5 召回 93% |

### 原则 5 — 事实核查优先于包装

包装前先核对真实数据来源——简历里每个数字都可能被面试官当场查证（点开 GitHub 看 README / data 文件 / 代码）。

- 项目有仓库 URL → 先 clone 或读，核对 README / report / data 里的真实数字
- 特别警惕「早期设想的数字」混进简历：计划评测 18 个模型 ≠ 代码真的评了 18 个。看 `data/*.csv` 实际行数
- 数字对不上就以**代码实际数据**为准，宁可写小而真（4 个模型）不要写大而假（18+）

**真实案例**：简历写「60 条标注 query」，但仓库 eval/RESULTS.md 实际是 15 条查询集——这种 4 倍偏差，面试官一对照就露馅。

### 原则 6 — 加粗样式：只加粗前缀，句中不加粗

```
\item \textbf{RAG 混合检索}：手写 BM25...，实测 Top-5 召回 93%
         ^^^^^^^^^^^^^^^ 加粗          ^^^^^^^^^^^^^^^ 不加粗
```

不要整条加粗（全强调=没强调），不要句中高亮（视觉碎片化）。

## 第二部分：命名与定位

### 原则 7 — 命名要查行业定义，尤其用「热门大词」时

用户常要「起个高大上的名字」。直接用热门词（AI Infra、平台、中台）风险高——这些词在业界有明确的技术范围期望，名不副实会在面试时露馅。

**操作方法**：
1. 用国内搜索引擎（Bing/百度，不要只靠 Google）搜该词的业界定义：大厂 JD、技术博客、白皮书
2. 对照用户的实际技能，判断覆盖了热门词定义的多大比例
3. 若只覆盖一小部分 → 加**限定词**收窄期望，而非硬蹭全称

**真实案例**：用户问「模型部署」能否叫 AI Infra。查大厂 JD 发现 AI Infra 核心是 C++/CUDA/分布式训练/GPU 集群，用户只做了端侧推理（MLX/llama.cpp/量化）。解法：不叫 AI Infra，叫「**端侧推理 Infra**」——加了「端侧」限定，既蹭到 Infra 的高大上感，又把面试官期望锁定在用户能力范围内。

给 3 个候选 + 推荐 + 理由，让用户选，不要直接定名。

## 第三部分：GitHub 仓库与简历联动

### 原则 8 — 仓库是简历的延伸，口径必须一致

面试官会从简历的 GitHub 链接点进仓库。如果简历写「自适应学习 Agent」，仓库却叫「实习技能树 / Skill_Tree」，README 标题是「🌳 实习技能树」——这种割裂会让面试官困惑项目到底是什么。

**联动检查清单**（简历改名时同步执行）：
- [ ] 仓库名（`gh repo rename`，旧 URL 自动 301 跳转）
- [ ] README 标题与开篇定位句
- [ ] GitHub description（`gh repo edit --description`）
- [ ] topics（`gh repo edit --add-topic/--remove-topic`，删与项目无关的、加精准的）
- [ ] git remote URL（`git remote set-url`）
- [ ] 简历里的 `\href{github链接}` 是否指向新仓库名

**README 结构**（技术展示导向，不是玩法介绍）：
1. 标题 + 一句话定位（专业，去 emoji 去学生气）
2. 核心能力（Agent 架构/RAG/Reflexion 等亮点上提）
3. 实测指标（把 eval 结果做成表格，面试官一眼看到工程含量）
4. 技术栈 + 快速开始 + 目录结构（保留）
5. 评估体系（强调可复现）

## 第四部分：公开内容净化

### 原则 9 — 公开仓库不能有「为面试而做」的动机痕迹

这是最容易被忽视的。commit history、文档措辞、代码注释、甚至 RESULTS.md 的标题，面试官都会看。任何暗示「这个项目是为了应付面试/求职而做的」措辞，都会让项目的真实性打折扣。

**三类要清除的措辞**：

| 类别 | 示例（要删） | 改法 |
|------|------------|------|
| **动机暴露** | 「包装为」「面试讲法」「可进简历」「面试应答手册」「可讲故事资产」「面试亮点」 | 改成中性技术描述（「重命名」「关键结论」「设计文档」） |
| **自证清白** | 「无虚构」「诚实记录」「真实记录」「不掺水」 | 删掉。用复现命令 + 原始数据路径代替自我担保——那才是真实性的硬证明 |
| **底气不足** | 「vibe coding 产物」「不可深讲」 | 改成正面描述（「精简技能栏，聚焦核心能力」） |

**为什么自证清白反而有害**：「无虚构」「诚实记录」这种词在对外文档里像在辩解，反而让读者起疑。好的技术报告不声明自己诚实——它用可复现的方法和数据说话，诚实是不证自明的。

**操作方法**：
- `git grep -n "面试\|讲法\|包装\|可进\|无虚构\|诚实\|vibe\|不可深讲"` 全局扫描
- 逐条判断：是动机暴露/自证清白 → 改；是正常功能语（如系统提示词里的「求职导师」、测试数据里的「面试准备」会话）→ 保留
- 整个文档都是面试导向的（如「面试应答手册」）→ 整个文件或目录移出公开仓库（gitignore + git rm --cached，本地保留）

**commit message 改写**：已 push 的 commit 若含暴露措辞，用 `git filter-branch --msg-filter` 批量改写 + `git push --force-with-lease`（个人仓库安全；用 `--with-lease` 比裸 `--force` 更安全）。改完清理 `refs/original/` 备份。

## 第五部分：LaTeX 简历的样式处理

当用户给的是 LaTeX 简历（`.tex`），除内容原则外：

- **高亮改加粗**：`\hlt{}`（黄底）→ `\textbf{}`，只在要点前缀用。
- **role 行右对齐**：`\role{技术栈}{右侧}` 用 `\hfill`。改 `.cls`：`{\par \textit{#1} \hfill #2 \par}`。右对齐后删掉 `\role` 第二参数开头的冗余 `\textbar\`。
- **溢出修复**：`\rightarrow` 用数学模式 `$\rightarrow$` 不可断行，连续多个会撑破行宽（Overfull hbox）。改用 UTF-8 箭头 `→` 可断行。
- **中英文空格**：中文与英文/数字之间加空格增强可读性（如「原生 function calling」而非「原生function calling」），但 `\href{}` 参数内、命令名后不加。
- **共享素材连锁**：`shared/experience/` 被所有 profile 引用，改一处全同步——这是特性，但要提醒用户其它 profile 也会变。
- **编译**：`xelatex -interaction=nonstopmode -halt-on-error`，中文必须 XeLaTeX；`TEXINPUTS` 注入 shared/ 和 profile 目录；用 `pdffonts` 查字体嵌入、`pdftoppm` 渲染查排版。
